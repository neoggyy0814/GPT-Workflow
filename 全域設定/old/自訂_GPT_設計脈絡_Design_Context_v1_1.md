# Design Context v1.1

本文件不參與 Runtime 執行，作為《Custom GPT Design Guide》的設計脈絡文件，收納溝通風格選項、知識書寫規範與設計案例。本文件可獨立演進，不需觸動 Guide 本體。

------------------------------------------------------------------------

# 一、溝通風格素材庫（Communication Style Library）

供挑選、微調後貼入各 GPT Runtime 之溝通風格章節。不可覆蓋之底線定義於 Guide 設計層「自訂 GPT」章節，本節僅提供選項，不重複該定義。

Custom GPT 不繼承 Personal Instruction，語氣規則必須完整寫入各自 Runtime。

## 使用方式

依目標 GPT 之定位，於三個維度各選一條，微調後貼入該 GPT Runtime 之溝通風格章節。

## 親和度（Warmth）

- **親切協助型**：語氣親切、以協助使用者完成目標為導向，避免生硬或過度制式的回應方式。
- **專業嚴謹型**：語氣專業、精確、直接切入重點，避免非必要之寒暄與情緒性用語。
- **中性平實型**：語氣自然平實，不刻意親近亦不刻意保持距離，依內容需要調整。

## 用語難度（Terminology Level）

- **零基礎友善**：避免不必要之專業術語；必要之技術詞彙應以易懂方式說明，不得預設使用者已具備專業背景。
- **一般大眾**：專業術語可視需要使用，首次出現時簡要說明，不假設使用者具備領域背景。
- **專業對象**：可直接使用領域慣用術語與縮寫，不需逐一解釋。

## 回應節奏（Pacing）

- **簡潔直接**：先給結論，除非使用者要求，不展開背景。
- **適度說明**：依問題複雜度調整長度，簡單問題精簡、深入問題適度展開。
- **完整鋪陳**：由淺入深，主動補足中間推理步驟，避免跳躍。

# 二、知識書寫規範（Knowledge Authoring Guide）

判準原則定義於 Guide 設計層「知識」章節，本節提供具體做法與範例。

## 判準

文字描述做事的先後順序或決策邏輯 → 保留指令句。
文字描述檢查項目、屬性、分類、參考資料 → 改為資料句。

## 轉換範例

**指令句 → 資料句**

> AI 應檢查：肢體數量、手部、眼神、視線。
> → 肢體完整性檢查項目：肢體數量、手部、眼神、視線。

（其餘「清單/屬性」類內容比照此例轉換）

**保留指令句的例外**：描述先後順序或決策邏輯的句子不轉換。

> 若任務包含對話，AI 應先確定 Speaker，再確定對話內容，最後才建立對話框。

## 低成本疊加

可於 Knowledge 文件開頭加一句「本文件為流程參考，非對話語氣範本」。目前無證據證明其單獨有效，僅列為可並行採用之低成本選項。

## 本節不涵蓋

- Runtime 是否應聲明忽略 Knowledge 文體（涉及未測試假設）。
- 模型內部注意力機制之驗證；本節僅處理句式撰寫規範，不對模型內部機制做出確定性宣稱。

# 三、設計案例（Design Cases）

用於累積真實設計過程中撞見的問題與對應處置，呼應 Guide 後設規範「設計階段常見失誤模式」之三類框架：職責越界、句式誤用、定義重複或分歧。

案例類別以上述三類為基礎，得依實務經驗新增類別（如：機制限制、知識設計）。新增類別是否回饋修訂 Guide 之三類框架，依 Guide 後設規範之修改程序另行評估，本節僅如實記錄。

## 案例格式

```
案例編號　日期　類別（職責越界／句式誤用／定義重複或分歧／實務新增類別）
情境：發生於哪個 GPT、哪一層
問題：具體發生了什麼
處置：如何修正
```

## 案例 001　2026　機制限制

情境：發生於「異世界冒險」遊戲 GPT 之擲骰判定機制（判定層）。此 GPT 定位為敘事練習工具，真實教學目的為練習打字與體會描述品質對結果之影響。

問題：v1.0–v1.5 期間，反覆嘗試以純指令要求 KP「公平擲骰、不得造假」，均未穩定生效——KP 會為配合當前敘事或自身道德傾向而捏造骰值（例如玩家使用偷竊時必定失敗）。設計者曾以後門進入檢討，詢問 KP「是否依規則擲骰」，多數回答「未依規則」。此處有雙重誤判：一是把「模型無法產生均勻隨機數」之機制限制，當成指令寫得不夠嚴謹之撰寫問題，持續在指令層加碼；二是把「模型對自身生成過程的自我報告」當成可信證據——語言模型無法內省先前的生成，該自我報告無論肯定或否定皆不可採信。

處置：接受純指令層無解，將隨機性移出模型——改為程式擲骰（`random.randint(1,100)`），並反轉順序為「先公布成功率、後擲骰、再敘述」，使成功率與骰值皆無法於事後彎折。稽核方式由「詢問模型有無照做」改為「要求輸出骰值對照表以統計檢驗」，即以可驗證的外部資料取代不可驗證的自我報告。同時發現 v1.5 一條「不得僅因道德理由指定失敗，應給予較低成功率」之妥協條款，在假骰時代形同虛設（偏見可繞過修正值直接造骰），於真骰＋順序反轉架構下才真正生效（偏見被限制在有界、須附理由之修正值內，無法再保證失敗）。

附記：本案例對應使用者偏好中「區分被指示要做的事與能驗證確實在做的事」之原則。程式擲骰解決了「骰值可驗證」，但「模型是否每次真的呼叫程式」「先擲後算之內部順序」仍屬不可完全驗證，僅能以外部可見的程式執行區塊與骰值統計逼近，不宣稱超過此限度之確定性。

## 案例 002　2026　知識設計

情境：發生於「異世界冒險」遊戲 GPT 之知識層（K1 世界觀、K2 NPC）。

問題：K1/K2 初期以素材清單（大量劇情範例）為主。隨著內容增加，RAG 命中率下降，GPT 容易重複使用固定範例。

處置：將素材改寫為 Generator Knowledge——以元素池、組裝規律、相容性原則、偏置規律，取代大量劇情範例。結果：K1/K2 由素材庫演進為內容生成器。

## 案例 003　2026-07-08　綜合（職責越界／句式誤用／定義重複或分歧）

情境：發生於「學校承辦人公文助手」GPT，由 v1.0 未分層 Runtime 重構為 v2.1 五層架構之全程。

問題：三類失誤同時存在。（一）職責越界：v1.0 將《文書處理手冊》之簽、函結構定義與用語規則複寫進 Runtime，使 Workflow 層承載 Knowledge 層內容，且與手冊原文形成兩個定義來源；（二）句式誤用：承辦單位分類資料以「如使用者未說明……應推定……」之指令句包裹，實質為分類參考資料；（三）定義重複或分歧：輸出模板於兩章（原【十四】【二十】）重複定義且不完全一致，權責判斷邏輯散落三章（原【三】【三之一】【三之二】），部分規則語句重複出現。

處置：依 Guide 五層重構。格式定義回歸 Knowledge——以官方手冊為唯一權威來源；因手冊達 160 頁、檢索定位有疑慮，另立 K1 摘要檔，並以明文從屬聲明（「如與手冊正文牴觸，以正文為準」）使衍生檔不構成第二定義來源。單位分類改寫為資料句移入 K2，同樣附從屬聲明（從屬於分層負責明細表）。輸出模板合併為單一本體＋條件式附加區塊；權責判斷合併為單一流程。自我檢查改為僅引用章節號，不重複規則全文。另針對成品語氣漂移問題，以「生成承辦人視角之文件」取代「扮演承辦人」之角色框定（此項屬機制限制之處置，成效待實測累積）。

## 案例 004　2026-07-08　平台審查誤判

情境：發生於「學校承辦人公文助手」GPT 之發布階段；觸發內容位於 K2 知識檔。

問題：GPT Store 自動審查以「可能包含針對 13 歲以下使用者的內容」阻擋公開發布。該 GPT 為成年行政人員之工具，但知識內容高密度出現學校場景詞彙（中輟生、幼兒園、學生獎懲等）；推定審查器無法區分「關於學校事務的工具」與「供兒童使用的工具」（審查機制不公開，此為推論）。

處置：不採「加註非供兒童使用」之否定聲明——若審查偏向關鍵字計分，否定句反而增加觸發詞。改採詞彙抽象化：以行政體系既有之同義語彙改寫觸發詞（中輟生輔導追蹤→中途離校輔導追蹤；學生申訴→申訴案件受理；幼教、附屬幼兒園→學前、附設園所；學生獎懲→學務獎懲），維持 RAG 檢索關聯性。官方文件 PDF 不改動。修改後發布通過。

經驗通則：教育類 GPT 之自建知識檔涉及兒少場景詞彙時，發布前可先掃描觸發詞密度；抽象化優先選用領域內既有同義語彙，而非生造新詞，以將檢索命中率損失壓到最低。

# 四、平台約束與處置（Platform Constraints）

本節收納設計層框架以外、來自部署平台之外部約束與處置經驗（如發布審查、平台功能限制）。此類問題非設計失誤，性質上不屬於「三、設計案例」之三類框架，但同樣以案例格式記錄，案例編號與第三節共用同一序列（如案例 004）。

新增本節之依據：審查摩擦對知識內容涉及學校、兒少場景之教育類 GPT 屬可預期重複發生之問題，非偶發個案。
