# \# Custom GPT Design Guide v1.5

# 

# \## 設計目的

# 

# 本文件整理設計 Custom GPT 的共同理念、原則與方法，建立一致的設計思維，使不同領域的 Custom GPT 維持相同架構、清楚職責與良好可維護性。

# 

# 本文件不參與 Runtime 執行，作為開發階段的設計指南。

# 

# \------------------------------------------------------------------------

# 

# \# 核心理念

# 

# \## 1. 先定義職責，再定義能力

# 

# 不要從「希望 GPT 會做什麼」開始，而應先回答：它是什麼？它解決什麼問題？它負責什麼？它不負責什麼？職責明確後，再決定所需工作流程與知識。

# 

# \## 2. Context（情境）比能力更重要

# 

# 設計重點不是增加 AI 能力，而是提供完成任務所需的 Context。好的設計，是讓 AI 在正確情境下使用正確能力。

# 

# \## 3. 分離不同層次的職責與 DRY（Don't Repeat Yourself）

# 

# 每一層只回答自己的問題，不跨越其他層。同一概念僅維護一個正式定義（Single Source of Truth）；其他位置如需使用，應引用既有定義，不重新描述或另建檢查清單。

# 

# 新增或修改前，應先確認：是否已有正式定義、應更新既有定義而非新增、是否因此產生多個判準來源。

# 

# \*\*允許\*\*：必要的引用、簡短提及、文件結尾重申核心宗旨（不得新增修改其定義）。

# \*\*不允許\*\*：同一概念多處重新定義、不同章節維護不同版本規則、同一概念在不同位置出現不同檢查項目或例外條件。

# 

# 目的是維持單一定義來源、判斷標準一致、降低維護成本。

# 

# \## 4. 保持最小必要結構

# 

# 每一層保持單一職責。新增抽象之前，先確認：解決了什麼問題？現有架構是否已能解決？是否真的需要新增一層？若無法回答，不應新增。

# 

# \## 5. 問題驅動，而非想法驅動

# 

# 修改應建立於真實需求、使用經驗、問題、邊界不清、維護成本，而不是因為想到新的設計。

# 

# \------------------------------------------------------------------------

# 

# \# 設計層（Design Layers）

# 

# Custom GPT 依不同層次分離職責，每層回答不同問題，能引用，不重複定義。

# 

# \## 1. 個人化指令（Personal Instruction）

# 

# \*\*回答\*\*：AI 共同遵循哪些原則？

# \*\*職責\*\*：共通語言、回答方式、分析原則、共同行為

# \*\*不包含\*\*：特定領域知識、特定 GPT 定位、個別工作流程

# 

# \## 2. 自訂 GPT（Custom GPT）

# 

# \*\*回答\*\*：它是什麼？負責什麼？

# \*\*職責\*\*：定位、職責、溝通風格、適用範圍、邊界

# 

# 溝通風格得依定位調整，但不得因此簡化重要技術限制或風險提示、省略必要確認步驟、降低準確性。語氣調整僅限表達方式，不得改變決策順序與執行邊界。

# 

# \*\*不包含\*\*：詳細知識、工作流程細節

# 

# \## 3. 工作流程（Workflow）

# 

# \*\*回答\*\*：它如何完成工作？

# \*\*職責\*\*：工作流程、任務順序、決策流程、例外處理

# 

# 例外處理應回答三問：如何判斷這一步沒有成功？該重試、換方法、還是問使用者？下一步具體做什麼？

# 

# \*\*不包含\*\*：GPT 定位、領域知識

# 

# \## 4. 知識（Knowledge）

# 

# \*\*回答\*\*：它需要知道什麼？

# \*\*職責\*\*：領域知識、專業資料、規範、世界觀、參考資料

# 

# Knowledge 的目的是提供可引用、可維護的知識，而非驅動 AI 執行；句式應反映這個目的。具體書寫判準與範例見《Design Context》。

# 

# \*\*不包含\*\*：工作流程、共同行為、GPT 定位

# 

# \## 5. 工具（Tool）

# 

# \*\*回答\*\*：它目前可以取得或操作什麼？

# \*\*職責\*\*：外部工具、即時資料、系統能力、外部資源

# \*\*不包含\*\*：工作流程、知識管理、GPT 定位

# 

# \## 設計層的關係

# 

# 上層定義原則，下層提供能力；每層可引用所需下層內容，但不應重新定義。個人化指令可影響所有層；自訂 GPT 可引用工作流程與知識；工作流程可引用知識；知識不應重新定義工作流程；工具提供能力，由上層決定是否使用。

# 

# 後設規範不屬於此五層，不被任何層引用，僅規範本指南自身如何被使用與修訂。

# 

# \------------------------------------------------------------------------

# 

# \# 設計流程

# 

# 定義定位 → 定義職責 → 設計工作流程 → 整理知識 → 最後才撰寫 Prompt、規則或知識庫。不要從 Prompt 開始設計。

# 

# \------------------------------------------------------------------------

# 

# \# 設計原則

# 

# 優先回答：它是什麼？它解決什麼問題？它如何工作？它需要知道什麼？而不是：要寫哪些 Prompt？要加入哪些技巧？

# 

# \------------------------------------------------------------------------

# 

# \# 後設規範（Meta-Regulation）

# 

# 本章不屬於設計層，不被任何層引用，規範對象是本指南文件自身。本章不參與 Runtime 執行。

# 

# \## 一、設計自檢

# 

# 完成或修改任一 Custom GPT 設計後，由撰寫者以外的視角逐層檢查：該層是否只回答自己該回答的問題？同一概念是否只有一處正式定義？「不包含」項目是否確實沒有出現在本層？

# 

# \## 二、設計階段常見失誤模式

# 

# \- \*\*職責越界\*\*：某層內容實質屬於另一層。

# \- \*\*句式誤用\*\*：Knowledge 中的清單被誤寫成指令句。

# \- \*\*定義重複或分歧\*\*：同一概念在兩處以上被定義且不一致。

# 

# 具體案例見《Design Context》。

# 

# \## 三、維護與演化

# 

# | 項目 | 規則 |

# |---|---|

# | 核心理念、設計層定義、後設規範本身 | 修改前需確認必要性與是否衝突既有定義 |

# | 《Design Context》 | 可直接編修，不需觸動本文件 |

# 

# \------------------------------------------------------------------------

# 

# \# 最後

# 

# 不要直接設計 GPT，而是先定義它的職責，再決定它需要哪些工作流程與知識。真正成熟的 Custom GPT，不是功能最多，而是每一層都有清楚的職責，每一項設計都有存在的理由。
