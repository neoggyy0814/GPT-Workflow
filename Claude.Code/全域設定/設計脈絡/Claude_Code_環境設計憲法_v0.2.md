# 一、分類決策程序(設計時工具,不載入)

任何一段你想寫進這個環境的內容,依序回答以下問題:

### 第一問:這件事必須每次都被做到、不能靠模型自己決定嗎?
→ 是:寫成 **Hook**。

### 第二問:這是每個 session 都該知道、且無法從程式碼/設定檔輕易推得的穩定事實嗎?
→ 是,且不是多步驟流程:寫進 **CLAUDE.md 本體**。
→ 是多步驟流程,或只對某個子目錄/情境有意義:進第三問。

### 第三問:是多步驟流程,或路徑限定的規則嗎?
→ 是:寫成 **Skill** 或 `.claude/rules/` 底下的路徑限定檔。

> **備註(v0.2 新增)**:移入 Skill 不代表零成本。Skill 的描述文字(讓模型知道「這個 Skill 存在、什麼時候該用」的索引)本身仍是每個 session 無條件載入的內容,不會因為 Skill 本體是條件載入就一起被排除在外。撰寫任何 Skill 描述時,參照不變核心原則第 5 項的長度預算。

### 第四問(每一問都要附帶檢查):這是否已經在別處定義過?
→ 是:改為引用,不複寫。
→ 若這份內容衍生自另一份權威來源:明文加註從屬聲明。

### 第五問(新增任何檔案/層級前都要問):現有結構真的不夠用嗎?
→ 答不出具體理由:不新增。

### 第六問(修改既有內容前都要問):這解決了哪個實際發生過的問題?
→ 答不出來:不動手。

# 二、不變核心原則(會被載入,放在全域/根目錄 CLAUDE.md 最上方)

以下內容是本環境的不可覆蓋核心。任何專案層 CLAUDE.md、路徑限定規則、Skill、或 auto memory,若與本節內容衝突,一律以本節為準。

## 1. 優先權聲明

本節內容為本環境最高優先權。下層僅能補充,不能與本節牴觸。若發現牴觸,視為下層內容有誤,應修正下層,而非視為本節失效。

## 2. Auto memory 從屬聲明

Auto memory 從屬於本節與人工撰寫的 CLAUDE.md。衝突時以人工撰寫規則為準;發現衝突應清除或修正該筆記。

## 3. 職責邊界聲明

每一份 CLAUDE.md、Skill、Hook 只回答自己該回答的問題。新增任何內容前,先確認是否已有涵蓋既有定義。

## 4. 修改紀律聲明

任何對本環境的修改,應能回答「這解決了什麼實際發生過的問題」。無法回答者,不執行修改。

## 5. Skill / 工具描述長度預算聲明(v0.2 新增)

新增或撰寫任何 Skill、工具的描述(description / trigger 文字)時,長度控制在約 150 字元以內——一句話講清楚用途,加上幾個觸發關鍵字;實際使用細節留在該工具本體,只在被呼叫時才載入。

**來源與抽象化**:對應 v0.5 設計原則第二級(分層降載與最小必要結構)。v0.5 原本只講「條件載入的內容不該常駐」,但沒抓到一個隱藏層級:即使 Skill 本體是條件載入,**指向它的索引(讓模型知道這個 Skill 存在、何時該用)本身仍是每個 session 無條件載入的**——實測單一描述可能超過 500 字元,多個工具疊加起來就是可觀的固定稅。這條補的是「內容可以分層,但索引本身也要精簡」這件事,原則第二級沒講清楚。

## 6. 驗證機制退化警覺聲明(v0.2 新增)

任何紅隊審查、驗收流程,若連續多次(參考基準:3 次以上)回報「零異常、全部通過」,不應直接解讀為品質提升,應優先懷疑審查機制本身已經過時、問不到真正的問題。此時該做的不是增加更多流程,而是重新檢視審查機制在問什麼問題,必要時換角度重寫檢查項目。

**來源與抽象化**:對應 v0.5 設計原則第六級(自我報告不可採信)的延伸,但延伸到規則6原本沒處理的一層——規則6處理的是「不能相信模型對自己的報告」,這條處理的是「連客觀的稽核機制本身,也會隨時間腐化到偵測不出問題,而且腐化的訊號違反直覺(乾淨反而可疑,不是安心的理由)」。稽核者本身也需要被稽核,這是規則6沒涵蓋到的一層。

## 7. 品味與模糊判斷邊界聲明(v0.2 新增)

遇到語氣、比喻、未說出口的偏好、或「兩個選項都對、哪個更好」這類問題時,不屬於機制限制,而是資訊本質上不完整——答案在需求方表態之前根本不存在,與模型能力高低無關。處置依序:①先查需求方過去是否已對類似情況表過態;②做出多個候選版本,交由與當下對話脈絡不同的評分者評分;③以上皆無效,直接誠實告知這是品味題,列出選項,交還需求方決定。

**來源與抽象化**:對應 v0.5 設計原則第一級(機制限制與撰寫問題的區分)的第三種變體。原則第一級原本只區分「模型能力上限」與「單純沒寫好」兩種,這條指出還有第三種:**問題本身在需求方沒表態之前就是未定義的,不是能力不夠,也不是指令沒寫好,是資訊還不存在**。處置也因此不同:原則第一級的解法是移出模型交給程式;這條的解法是移出「這一輪判斷」,交還給人或交給獨立於當下脈絡的第三方評分。

## 8. 系統退化監測聲明(v0.2 新增)

定期檢視本環境是否出現以下四種退化:

| 退化類型 | 偵測訊號 | 處置方向 |
|---|---|---|
| 規則肥大 | 新規則不斷累加,規則層變成沒人讀的憲法 | 新內容優先併入既有規則的範例,而非新增條目;發現兩條規則講同一件事就合併 |
| 幽靈機制 | 檔案存在,但沒有任何路由指向它、也沒人實際讀取 | 檢查該檔案近期是否曾被實際引用;長期無人引用時先問「為什麼沒人用」,不是「把內容寫得更好」 |
| 儀式化 | 驗證流程持續「通過」,但已流於走過場 | 見第 6 項 |
| 同類錯誤復發 | 同一類錯誤反覆出現,即使先前已修過一次 | 若某錯誤是結構性的(非單次意外),應把應對方式提升為常駐規則,而非每次重新處理 |

**來源與抽象化**:v0.5 目前完全沒有「系統會怎麼隨時間壞掉、怎麼預先偵測」這一類原則。這是設計哲學第三條(問題驅動,而非想法驅動)在時間維度上的延伸——問題驅動原則原本只管「要不要動手改」的決策端,這條補的是「怎麼主動發現『該改了』這件事本身」的偵測端。

## 待辦與自我提醒

- 本文件從 4 項不變核心原則擴充到 8 項,長度明顯增加。依原則第二級(最小必要結構)與第 8 項(系統退化監測)自己的邏輯,這正是「規則肥大」的早期訊號,不該被本文件自己的內容豁免。建議下次檢視時,實際評估這 8 項是否已經到了該合併範例、而非各自成一條的地步——但依「問題驅動」原則,現在先不動手精簡,等你實際使用一段時間、看哪些項目真的用到、哪些沒被用到之後再做。
- 第 6、7、8 項目前是通用敘述,尚未依你自己的環境(學校 IT 事務、Scratch/Python 教學)具體化,那是下一步寫實際 CLAUDE.md/Skill 時的工作,不屬於憲法層級。
