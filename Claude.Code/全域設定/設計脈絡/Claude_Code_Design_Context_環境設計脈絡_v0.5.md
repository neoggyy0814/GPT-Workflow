本文件不參與 Runtime 執行,作為《Claude_Code_Constitution_環境設計憲法》的設計脈絡文件,收納版本異動歷程、每條原則的來源案例與完整推理。本文件可獨立演進,不需觸動憲法本體。

沿用《自訂 GPT 設計脈絡》既有的案例格式,不重新發明:

```
案例編號　日期　類別
情境:發生於哪裡
問題:具體發生了什麼
處置:如何處理
```

# 二、設計案例(Design Cases)

## 案例001　2026-07-09　分層降載延伸

情境:整理他人的 Claude Code ops 部署案例(`50-diagnosis.md` Leak 1)時發現。

問題:即使 Skill 本體是條件載入,指向它的描述索引(讓模型知道這個 Skill 存在、何時該用)本身仍是每個 session 無條件載入的內容——實測單一描述可能超過 500 字元,多個工具疊加即成為可觀的固定稅。通用指南 v0.5 設計原則第二級(分層降載與最小必要結構)原本只講「條件載入的內容不該常駐」,沒抓到「內容可以分層,但指向內容的索引本身也要精簡」這一層。

處置:訂為憲法不變核心原則第5項——新增或撰寫任何 Skill、工具的描述文字,控制在約 150 字元以內,細節留在工具本體。

## 案例002　2026-07-09　自我報告不可採信延伸

情境:整理 `60-letter.md`「Degradation 3:儀式化」段落時發現。

問題:原始案例記載,紅隊審查若連續多次(該部署真實基準:連續4次)回報零異常,不是品質變好的證據,而是審查機制本身已經過時、問不到真正的問題——乾淨的連續紀錄反而是異常,不是常態。通用指南 v0.5 規則6(自我報告不可採信)處理的是「不能相信模型對自己的報告」,但沒處理「連客觀的稽核機制本身,也會隨時間腐化到偵測不出問題」這一層,且腐化訊號違反直覺。

處置:訂為憲法不變核心原則第6項——驗證機制連續零異常時,優先懷疑機制本身過時,而非解讀為品質提升。

## 案例003　2026-07-09　機制限制的第三種變體

情境:整理 `20-judgment.md` R6(品味題與模糊判斷)時發現。

問題:通用指南 v0.5 規則1(機制限制與撰寫問題的區分)原本只區分「模型能力上限」與「單純沒寫好」兩種情況。R6 揭示了第三種:語氣、比喻、未表態的偏好、「兩者皆對誰更優」這類問題,答案在需求方表態之前根本不存在——不是模型能力不夠,也不是指令沒寫好,是資訊本身尚未產生。處置方式也因此不同於規則1(移出模型交給程式):這裡的處置是移出「這一輪判斷」,交還給人或交給獨立於當下脈絡的第三方評分。

處置:訂為憲法不變核心原則第7項,依序:①查需求方過去表態→②多候選交獨立評分者→③誠實列選項交還需求方。

## 案例004　2026-07-09　問題驅動原則的偵測端

情境:整理 `60-letter.md`「這整套系統最可能怎麼退化」段落時發現(規則肥大、幽靈機制、儀式化、同類錯誤復發四類)。

問題:通用指南 v0.5 設計哲學第三條(問題驅動,而非想法驅動)原本只管「要不要動手改」的決策端,完全沒有「怎麼主動發現『該改了』這件事本身」的偵測端。

處置:訂為憲法不變核心原則第8項,列出四類退化的偵測訊號與處置方向:

| 退化類型 | 偵測訊號 | 處置方向 |
|---|---|---|
| 規則肥大 | 新規則不斷累加,規則層變成沒人讀的憲法 | 新內容優先併入既有規則的範例,而非新增條目;兩條規則講同一件事就合併 |
| 幽靈機制 | 檔案存在,但沒有任何路由指向它、也沒人實際讀取 | 檢查該檔案近期是否曾被實際引用;長期無人引用時先問「為什麼沒人用」,不是「把內容寫得更好」 |
| 儀式化 | 驗證流程持續「通過」,但已流於走過場 | 見案例002 |
| 同類錯誤復發 | 同一類錯誤反覆出現,即使先前已修過一次 | 若錯誤是結構性的(非單次意外),應把應對方式提升為常駐規則,而非每次重新處理 |

## 案例005　2026-07-09　文件治理(憲法自身的規則肥大)

情境:發生於《Claude Code 環境設計憲法》本身。v0.1 → v0.2 因新增四項不變核心原則(當時含完整來源與抽象化說明)而長度明顯增加。

問題:v0.2 的憲法同時承載兩種性質不同的內容——穩定、需要每次載入的核心原則本身,以及只在設計/回顧時才需要的版本歷程、案例來源、詳細推理。這正是憲法自己第8項「系統退化監測聲明」裡「規則肥大」的訊號,而且是由憲法自己的內容產生的,不是外部案例。

處置:仿照《自訂 GPT 設計指南》與《自訂 GPT 設計脈絡》原有的分工,將憲法拆分為兩份文件——憲法只留分類決策程序與不變核心原則的條文本身;版本異動、案例來源、完整推理移入本文件(即本案例所在的文件)。

附記:這次拆分本身,是通用指南 v0.5 設計原則第二級(分層降載與最小必要結構)遞迴套用在治理文件自己身上的示範——不只用來設計「給 AI 讀的內容」,也適用於「人類自己維護、用來設計 AI 內容的文件」。

## 案例009　2026-07-09　執行既有規則與新增規則的確認門檻區分

情境:整理憲法時,主動清除條文內殘留的版本標籤(見案例008附記),事後才向使用者說明理由,未事先徵詢。使用者認可這個做法,並指出這是他個人風格偏好中重要的一條。

問題:通用指南設計哲學第3條(問題驅動,而非想法驅動)原本沒有區分「修正既有已確立規則的明顯違反」與「新增未曾確立的原則」——前者其實不需要事先徵詢(規則已經同意過,只是執行/落實),後者才需要先提出、取得確認。混在一起容易造成兩種錯誤:該直接修正的地方猶豫詢問、或該先問的新想法逕自併入。

處置:於通用指南設計哲學第3條、憲法不變核心原則第4項(修改紀律聲明),補上這個區分:執行既有規則可逕行處理;新增規則須先提出、取得確認。

## 案例010　2026-07-09　中斷風險價值排序與資源將盡收尾

情境:使用者提供第二篇獨立文章(關於長時任務作業規則),其中「先做價值最高的產出、隨時可能中斷、落檔內容即為使用者拿到的全部」的做法,經使用者確認是他本人在實際使用額度限制時真實撞過牆的經驗,不是單純的理論主張。同一份文章也提出「資源將盡時應停止產出、優先完成收尾與交接」,使用者認為重要但當下不知如何具體落地。

問題:通用指南既有的排序邏輯只處理「依相依關係決定平行/序列」(源自 `00-command-loop.md` Step 4),沒有處理「當存在中斷風險時,該按什麼順序執行」這個不同維度的問題。「資源將盡」本身也需要具體的觸發訊號與對應動作,否則只是一句無法執行的空話。

處置:於通用指南新增第八條設計原則「中斷風險下的價值排序」,與依賴關係排序並存、不衝突。於憲法新增不變核心原則第10項,將觸發訊號具體化為:用量達約 80%,或環境提供的顯式提示(系統提示顯示剩餘量偏低、自動壓縮已發生一次以上);若無顯式訊號則退回行為代理指標(對話輪數明顯超出預期)。觸發後的動作定為三步:落檔已完成內容→簡短總結現況→寫入交接紀錄。

附記(擱置案例,未採納):同一篇文章亦提出「弱模型需要明確、強模型需要留白」,主張具體程度應對應執行者能力層級。使用者誠實表示自己使用通用模型、任務規模也未觸及多層級模型派工,無法從自身經驗驗證這件事,依問題驅動哲學暫不收錄,僅記錄於通用指南待辦供未來參考。此案例值得注意之處在於:一個抽象論證站得住腳,不代表現在就該收進使用者自己的憲法——兩者是不同層次的判斷,不應混淆。

## 案例006　2026-07-09　新增類別:建設性發散原則

情境:使用者在對話中觀察到,ChatGPT 常需要被收斂發散思考,而 Claude 相對缺乏主動提出「使用者沒想到、但重要」的方向。

問題:通用指南 v0.5(當時)六條設計原則都在處理「AI 怎麼正確處理已經給它的內容」,沒有任何一條處理「AI 該不該主動提供內容之外、但重要的東西」。這條的證據基礎與案例001–004不同——不是第三方部署的具體日期事件,而是使用者本人在本次對話過程中的當下觀察,樣本數為一,合理但非可重現的部署紀錄,如實標注,不假裝證據力相同。

處置:於通用指南 v0.6 新增第七條設計原則「建設性發散原則」,不參與控制強度排序(問的是不同軸線的問題);於憲法 v0.4 新增不變核心原則第9項。條文加上明確限定詞(只提重要且方向明確的項目、簡短呈現、不能取代直接回答),以避免退化成無差別發散,呼應問題驅動哲學「發散要有具體理由」的精神。

附記:本案例本身也是這條原則的即時示範——回答時同步指出「憲法本身尚未經過 living proof 檢驗」這個使用者當下未問、但重要且方向明確的觀察。

## 案例007　2026-07-09　建設性發散原則的適用範圍修正

情境:使用者澄清自己 userPreferences 裡「前提無誤時直接產出結果,不重複說明」這條的原意,是針對**分析產出階段**(不要重複討論分析過程本身,直接給結果),不是全程禁止主動提出未竟之處;**討論/構思階段**(如本次對話)則希望主動被提醒沒看到的重要部分。

問題:案例006新增的建設性發散原則,條文沒有區分「討論階段」與「分析產出階段」,可能被誤套到使用者明確要求乾淨分析結果的時候,反而在結果前後插入雜訊,與使用者原本的偏好意圖衝突。

處置:不新增條目,直接修正原條文(通用指南第七條、憲法第9項),補上「適用範圍依任務性質區分」:討論/構思階段適用;使用者明確要求直接分析產出時,先給出直接完整結果,建設性發散若有,以簡短附註接在結果之後,不得取代或延遲結果本身。此修正遵守最小必要結構——修正既有條文,不因為一次澄清就疊加新規則。

## 案例008　2026-07-09　可執行程度軸線:第二來源出現,正式收錄

情境:整理 Claude Code ops 案例(`00-command-loop.md`、`10-dispatch.md`)時,曾發現「指令句即使性質正確,若抽象到無法直接執行,實務上等同沒寫」這件事,當時列為候選但未收錄。後續使用者提供另一篇獨立、不相關的通用提示工程文章,文章中「具體化要求」「給範例(few-shot)」「指定輸出格式」三點,合起來講的正是同一件事。

問題:通用指南第三級「指令句與資料句」原本只處理「性質軸線」(指令/資料),沒有處理「具體程度軸線」(可執行/抽象)。此軸線首次出現時只有單一部署案例佐證,這次由完全獨立的第二來源再次指出,證據結構從「單一案例」升級為「兩個互不相干來源交會」,證據力高於第一次提出時。

處置:修正通用指南第三級原條文(不新增條目),拆為軸線一(性質)、軸線二(具體程度)兩個依序檢查的子判準,並註明兩者的先後關係——軸線一決定資格,軸線二決定力道。同步在憲法分類決策程序第四問加入對應檢查點。

附記(版本標籤清理):同一次修訂中,發現憲法 v0.4–v0.5 條文內殘留「(v0.2 新增)」「(v0.4 新增)」等版本標籤,這違反 v0.2→v0.3 拆分時定下的規矩——版本歷程只該留在設計脈絡,憲法本體應為乾淨的結論狀態。此次一併清除,不算新案例,是拆分工作未徹底執行的殘留修正。
