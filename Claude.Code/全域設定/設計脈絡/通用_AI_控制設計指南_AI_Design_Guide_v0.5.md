# 通用 AI 控制設計指南 v0.5

本文件收斂自「自訂 GPT 設計指南／設計脈絡」之實務案例,抽取其中不限定於單一平台的通用原理,供跨平台(Custom GPT、Claude、其他 LLM 應用)的 AI 控制設計參考。

## 版本異動(v0.4 → v0.5)

- 整份文件重整為兩部分:**設計哲學(Design Philosophy)**與**設計原則(Design Principles)**。原「設計流程紀律」章節併入設計哲學,原「控制規則」正名為設計原則,結構不變、名稱與定位重新釐清。
- 設計哲學新增第三項:**脈絡比能力更重要(Context Matters More Than Capability)**。
- 設計哲學內部改依「設計時序」排序(動手前 → 動手時 → 上線後),不使用控制強度,因三項哲學對模型輸出的直接支配力皆為零。
- 明文標出每項設計原則對應到哪項設計哲學,呈現「哲學生成原則」的關係,而非兩份互不相干的清單。

---

## 兩者的區分

- **設計哲學**:作用對象是**設計者**,發生在任何內容寫進 Prompt、CLAUDE.md、Runtime、Project 指示**之前**。回答「我該用什麼心態、什麼優先順序來做這件事」。對模型輸出沒有直接支配力——它決定的是你會不會走到需要設計原則發揮作用的那一步,以及走到那一步時你會怎麼選。
- **設計原則**:作用對象是**模型**,是實際會被 AI 讀取、處理的內容所依循的規則。回答「這段內容該怎麼寫,AI 才會正確地讀」。可以用「控制強度」排序,因為它們直接影響模型輸出。

判斷一項屬於哲學還是原則,問自己:**拿掉這一條,模型的輸出行為會不會改變?** 若不會(只有設計者的決策過程改變),屬於哲學;若會,屬於原則。

---

# 一、設計哲學(Design Philosophy)——依設計時序排列

## 1. 職責先於能力(Responsibility Before Capability)

**動手前該想的事。** 動手寫任何控制內容之前,先回答:它是什麼?它解決什麼問題?它負責什麼?它不負責什麼?職責明確後,才決定需要的工作流程;工作流程確認後,才整理所需知識;最後才撰寫 Prompt、規則或知識庫內容。不要從「希望 AI 會做什麼」開始。

**通用性依據**:一般系統設計紀律(先釐清需求與範圍,再實作),非模型機制決定。但 AI 控制設計有特有的風險放大因素——寫一段自然語言指令直接丟給模型的門檻極低,不像寫程式會被編譯器或測試擋下明顯的架構錯誤,所以「跳過釐清職責、直接動手寫指令」在這個領域特別容易發生,且不會立刻報錯,只會在後續維護時才顯現代價(層次越界、定義分歧)。

## 2. 脈絡比能力更重要(Context Matters More Than Capability)

**動手時,決定每一層放什麼內容的事。** 設計重點不是增加模型能力(換更強的模型、加更多技巧),而是提供完成任務所需的脈絡。好的設計,是讓模型在正確的脈絡下使用正確的能力。

**通用性依據**:此項有直接的機制根據,強度高於一般工程紀律。模型的能力(推理、知識、技能)已固化於權重,同一模型面對同一任務,輸出品質的差異主要來自當下 context window 裡有什麼、如何排列優先順序,而非切換更聰明的模型——這是目前 AI 領域已獨立成科的實務主題(context engineering),不是設計者的經驗歸納。

**生成關係**:本項是設計原則第一級(機制限制與撰寫問題的區分)診斷步驟的理論基礎——遇到表現不佳,先問「脈絡給對了嗎」,只有補足脈絡仍無效時,才能下「已達機制限制」的結論。也是設計原則第二級(分層降載與最小必要結構)的上游理由——之所以要精心管理內容的載入時機與結構精簡度,正是因為脈絡的品質比模型能力本身更決定輸出品質。

## 3. 問題驅動,而非想法驅動(Problem-Driven, Not Idea-Driven Revision)

**上線後,決定要不要改的事。** 任何對控制內容(Runtime、CLAUDE.md、系統提示、Project 指示等)的修改,應建立於真實需求、使用經驗、觀察到的問題、邊界不清、或維護成本——而不是「想到一個更完美的概念」。修改前可自問:這項修改解決了什麼具體發生過的問題?若答不出來,不應修改。

**通用性依據**:同樣是一般工程紀律,非模型機制特有。放大因素在於:自然語言規則沒有編譯期檢查或測試套件能自動偵測「這個新想法破壞了什麼」,想法驅動的修改造成的副作用只會在模型行為飄移時才被觀察到,且往往難以追溯是哪一次修改造成的。

# 二、設計原則(Design Principles)——依控制強度排序

「控制強度」定義為:**這條規則對模型實際輸出的支配力有多強、多不可繞過**。排序從「一旦不成立,其他規則寫得再好也無法生效」的一端,排到「不直接改變模型輸出、僅影響設計者能否信任驗證結果」的一端。

判斷通用性的標準:這條規則描述的是語言模型作為一種生成機制的共同屬性,還是某個平台的介面設計選擇?前者跨平台通用,後者不應直接移植。

## 第一級:機制限制與撰寫問題的區分(Model Capability Limits vs. Prompt-Engineering Problems)

*對應哲學:脈絡比能力更重要(診斷步驟的理論基礎)*

控制強度最高,因為它是唯一的「硬邊界」,一旦碰到,其餘規則寫得再精確也無法生效。當某項要求(如保證真隨機性、保證長文件全域一致)在反覆修改指令後仍不穩定,應先判斷是否已達模型生成機制的邊界——依「脈絡比能力更重要」的哲學,先確認是否為脈絡不足導致,而非模型能力極限。若確認已達邊界,純指令層無解,應將該部分移出純語言生成(例如改用程式執行,或如 Claude Code 的 Hook 機制,把決定權交給不經過模型判斷的外部程式)。

**通用性依據**:任何模型都有無法單靠加強指令語氣解決的能力邊界,差別只在邊界位置,不在有沒有邊界。這是硬邊界:碰到即無解,不是機率問題。

## 第二級:分層降載與最小必要結構(Tiered Loading & Minimal Necessary Structure)

*對應哲學:脈絡比能力更重要(上游理由)*

控制強度僅次於硬邊界,因為它是「軟邊界」:不是碰到就完全失效,而是隨內容量與結構複雜度增加,其餘規則的實際效果會被逐漸稀釋。

- **分層降載**:不是所有內容都該以同樣的優先權、同樣的常駐程度放進同一個容器。應依「每次互動是否真的需要」區分:穩定核心無條件常駐,情境相關內容依需要才載入,必須每次都做到的交給外部強制機制(如 Hook)而非仰賴模型自律。
- **最小必要結構**:新增分層、文件、抽象層之前,先確認是否真的解決了問題,現有結構是否已足夠。

**通用性依據**:內容量與結構直接影響模型的注意力分配與檢索精準度,這是 transformer 架構本身的屬性,不只是「維護成本更高」而已。實證可見於長期知識庫內容增加後,檢索命中率下降、模型開始重複使用固定內容的現象。

## 第三級:指令句與資料句(Imperative Phrasing vs. Declarative Phrasing)

在「可透過指令控制」的範圍內,這是最基礎、最直接的控制槓桿,因為它決定模型是否把一段文字當成「該執行的指令」還是「該參考的資料」——這是後面兩條規則能否被正確讀取為「規則」的前提。

**通用性依據**:所有基於 next-token prediction 訓練的模型解析 prompt 的共同方式,不是特定平台的解析邏輯。

## 第四級:單一事實來源(Single Source of Truth / DRY)

控制的是「多處規則之間會不會互相打架」,強度低於前三條——即使每處定義的語氣、句式都正確,結構也夠精簡,只要出現兩處不一致的定義,模型仍可能任意選用其中一版。

**通用性依據**:資訊系統設計的通用原理,不限 AI 語境,只是在 AI 語境下代價更高。

## 第五級:從屬聲明(Subordination Clause with Fallback)

第四級的局部強化版,適用範圍更窄:專門處理「衍生檔案 vs 權威來源」結構下的定義分歧風險。

**通用性依據**:多文件系統的通用做法,與模型廠商無關。

## 第六級:自我報告不可採信(Unreliable Self-Report)

控制強度最低,因為它不改變模型的輸出行為本身,只改變設計者「能不能信任驗證結果」。

**通用性依據**:現行 transformer 架構的推論本質決定了模型沒有對「先前生成過程」的內省管道,與廠商無關。

---

## 不列入的項目(平台特定,不應直接移植)

- **文體獨立性 / 文體污染**:源於「Custom GPT 平台不繼承 Personal Instruction」之特定架構缺陷。
- **平台審查行為特性**:審查機制內部邏輯不公開,任何「觸發詞規律」皆屬推論。
- **具體的分層架構設計(如五層 Custom GPT 架構)**:「分層與精簡」的原則本身通用,但「該分成幾層、每層叫什麼名字」是平台相依的具體設計。

## 在 Claude 上的適用性摘要

### 設計哲學

| 項目 | Claude 適用性 |
|---|---|
| 職責先於能力 | 適用於設計 Project 指示或 CLAUDE.md 之前的準備工作,與平台無關。 |
| 脈絡比能力更重要 | 適用。決定個人偏好指示、Project 指示、Project 知識庫、CLAUDE.md 各放什麼內容時的核心判準。 |
| 問題驅動,而非想法驅動 | 適用於任何修改 Project 指示、CLAUDE.md、個人偏好指示的時機。 |

### 設計原則

| 級別 | 規則 | Claude 適用性 |
|---|---|---|
| 1 | 機制限制與撰寫問題 | 適用。Claude.ai 沒有 Hook 機制,但有程式碼執行工具;Claude Code 有原生 Hook。 |
| 2 | 分層降載與最小必要結構 | 適用。對應 Claude.ai 的「個人偏好指示 / Project 指示 / Project 知識庫」三種容器,以及 Claude Code 的「CLAUDE.md / Skill / Hook」三層。 |
| 3 | 指令句 / 資料句 | 適用。上傳文件、貼上的參考資料若含祈使句語氣,可能被誤讀為主動指令。 |
| 4 | 單一事實來源 | 適用。同時使用 Project 指示、Project 知識庫、個人偏好指示時需注意重疊。 |
| 5 | 從屬聲明 | 適用。Project 知識庫中同時放摘要檔與原始文件時建議加註從屬關係。 |
| 6 | 自我報告不可採信 | 適用,且對「Claude 解釋自己如何運作」這件事本身同樣成立。 |

## 待辦(後續版本可處理)

- 檢視原 Guide 核心理念是否還有其他項目未覆核(目前五項核心理念已全數處理:職責先於能力、脈絡比能力更重要、分層/DRY 併入設計原則第二、四級、最小必要結構併入設計原則第二級、問題驅動)。
- 依「發生頻率」或「修正成本」等其他軸線重新排序設計原則,比較與本版「支配力」排序的差異。
- 補充 Claude Code 的 Hook 事件分類作為設計原則第一級的具體案例。
