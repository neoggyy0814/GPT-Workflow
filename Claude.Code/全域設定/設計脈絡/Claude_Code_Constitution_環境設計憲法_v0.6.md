# 一、分類決策程序(設計時工具,不載入)

任何一段你想寫進這個環境的內容,依序回答:

1. **這件事必須每次都被做到、不能靠模型自己決定嗎?** → 是:寫成 **Hook**。
2. **這是每個 session 都該知道、且無法從程式碼/設定檔輕易推得的穩定事實嗎?** → 是,且不是多步驟流程:寫進 **CLAUDE.md 本體**。是多步驟流程或只對特定情境有意義:進第 3 問。(撰寫 Skill 描述時參照不變核心原則第 5 項的長度預算。)
3. **是多步驟流程,或路徑限定的規則嗎?** → 是:寫成 **Skill** 或 `.claude/rules/` 路徑限定檔。
4. **這是否已經在別處定義過?** → 是:改為引用,不複寫。若衍生自另一份權威來源,明文加註從屬聲明。撰寫任何規則/Skill 內容時,同步檢查兩個獨立軸線:這是指令還是資料?若是指令,具體到能被直接執行嗎(有無正反範例、可檢驗的輸出格式)?兩者缺一,內容效力都會打折扣。
5. **現有結構真的不夠用嗎?**(新增任何檔案/層級前都要問)→ 答不出具體理由:不新增。
6. **這解決了哪個實際發生過的問題?**(修改既有內容前都要問)→ 答不出來:不動手。

# 二、不變核心原則(會被載入,放在全域/根目錄 CLAUDE.md 最上方)

以下內容為本環境最高優先權。下層僅能補充,不能牴觸。牴觸時,視為下層有誤,修正下層。

1. **優先權聲明**:本節優先於一切下層內容。
2. **Auto memory 從屬聲明**:Auto memory 從屬於本節與人工撰寫的 CLAUDE.md;衝突時以人工規則為準,並清除或修正衝突的筆記。
3. **職責邊界聲明**:每一份 CLAUDE.md、Skill、Hook 只回答自己該回答的問題;新增前先確認是否已有涵蓋。
4. **修改紀律聲明**:任何修改應能回答「解決了什麼實際發生過的問題」,答不出來不執行。
5. **Skill / 工具描述長度預算聲明**:新增或撰寫任何 Skill、工具的描述文字,控制在約 150 字元以內;細節留在工具本體,只在被呼叫時載入。
6. **驗證機制退化警覺聲明**:紅隊審查或驗收流程連續多次(參考基準:3 次以上)零異常,不解讀為品質提升,應優先懷疑審查機制本身已過時。
7. **品味與模糊判斷邊界聲明**:語氣、比喻、未表態的偏好、「兩者皆對誰更優」不屬於機制限制,是資訊本質上不完整。依序:查過去表態→多候選交獨立評分者評分→誠實列選項交還需求方。
8. **系統退化監測聲明**:定期檢視是否出現規則肥大、幽靈機制、儀式化、同類錯誤復發——四類皆有對應偵測訊號與處置方向,詳見設計脈絡。
9. **建設性發散聲明**:完成任務時,若發現使用者未曾提及、但具體且對結果有正面影響的方向,應主動簡短提出,而非只收斂在原問題範圍內作答。適用範圍依任務性質區分:討論/構思階段適用;使用者明確要求直接的分析產出結果時,先給出直接、完整的結果,不重新討論分析過程本身,建設性發散若有,以簡短附註形式接在結果之後,不得取代或延遲結果本身。只提重要且方向明確的項目,不列舉無差別替代方案;以簡短提示呈現,不展開成新分析除非被要求。
